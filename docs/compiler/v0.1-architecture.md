# XLang v0.1 Compiler Architecture

Architecture of the bootstrap compiler in `compiler/` (Rust crate **`x`**, version **0.1.0**).

For language rules see [v0.1 language reference](../spec/v0.1-language-reference.md). For release scope see [v0.1 release notes](../releases/v0.1.md).

---

## Goals

| Goal | v0.1 approach |
|------|----------------|
| Determinism | Single-pass frontend, pinned IR snapshots in tests |
| Safety gate | `Module::verify()` before any IR leaves the compiler |
| Simplicity | Direct LLVM lowering ŌĆö no C IR stage |
| Editor support | Shared analysis in `lsp.rs`, consumed by `LSP/` |

## Frontend

### Lexer (`lexer.rs`, `token.rs`)

- Scans UTF-8 source byte-by-byte with line/column tracking.
- Emits `Token { kind, span }` for identifiers, literals, keywords, operators.
- Skips whitespace and comments (`//`, `/* */`, `///`).
- Rejects UTF-8 BOM and unknown characters.
- Future keywords are reserved at lex time; `fn`, `let`, `var` are **identifiers**.

### Parser (`parser.rs`)

- Builds `Program` from token stream.
- Top level: optional `module`, imports, structs, functions.
- Functions: C-style `type name(params) { stmts }`.
- Statements: bindings, assign, return, if/else, expression statements.
- Expressions: Pratt parser for operator precedence; calls, literals, variables.
- Produces parse diagnostics with spans on failure.

### AST (`ast.rs`)

Core nodes:

| Node | Purpose |
|------|---------|
| `Program` | Root: module, imports, structs, functions |
| `Function` | Signature + body block |
| `StructDecl` | Name + typed fields |
| `Stmt` | Let binding, assign, return, if, expr stmt |
| `Expr` | Literals, variable, call, unary, binary |
| `TypeName` | `I32`, `Bool`, `Str`, `Void`, `Named(String)` |

Internal `Stmt::Let` represents **binding statements** (`i32 x = ŌĆ”`), not a `let` keyword.

### Type checker (`typeck.rs`)

Single pass over each function after collecting function signatures.

Checks include:

- Duplicate function, struct, field, parameter, and local names
- Supported signature types (`i32`, `bool`, `void`, `str`; named types rejected in signatures)
- Expression typing and operator rules
- Assignment mutability and type equality
- Call arity and argument types
- `main` signature (`i32 main()`)
- Definite return for non-void functions
- Unreachable statements
- Literal `i32` range and division by zero
- Branch-local visibility rules

Returns `XResult<()>` ŌĆö first error stops the pipeline.

## Backend

### LLVM emitter (`backend/llvm.rs`)

Uses [Inkwell 0.9](https://github.com/TheDan64/inkwell) targeting LLVM 22.

Flow:

1. Reject unsupported constructs (`str`, struct values, etc.)
2. Declare all functions with LLVM types
3. Emit each function body into an `entry` block
4. Lower locals to `alloca` + `store` / `load`
5. Lower control flow: `if`/`else` with basic blocks; short-circuit `&&`/`||`
6. Run `Module::verify()` before returning IR string

Supported value types: `i32`, `i1` (bool), void returns.

### Build integration (`compile.rs`)

| Step | Tool |
|------|------|
| Write IR | `build/main.ll` |
| Link | `clang -Wno-override-module [-target ŌĆ”] -o build/main.exe build/main.ll` |

`CompileOptions::target_triple` flows from CLI or `XLANG_TARGET_TRIPLE`.

## Diagnostics (`diagnostic.rs`)

```rust
pub struct Diagnostic {
    pub code: DiagnosticCode,  // Lexical | Parse | Type | Backend | Io | Internal
    pub message: String,
    pub span: Span,
}
```

Spans track byte offsets and line/column ranges for LSP and CLI output. Stable `E-*` code strings are assigned at reporting sites.

## LSP integration (`lsp.rs`)

Shared analysis for the language server:

```text
analyze_source(source) ŌåÆ AnalysisResult {
    program: Option<Program>,
    diagnostics: Vec<Diagnostic>,
    index: SemanticIndex,
}
```

`SemanticIndex` maps symbols (functions, params, locals, structs, types) and references (reads, calls). Used for hover markdown, completion, definition, references, rename, and semantic tokens.

The LSP crate (`LSP/`) depends on `x` and implements the tower-lsp protocol in `LSP/src/main.rs`.

---

## CLI (`main.rs`)

```text
x [check|emit-llvm|build|run] [options] [file.x]
```

- Default file: `main.x` in the working directory
- Options: `--target`, `--target=<triple>`

Exit code: `0` on success; non-zero on diagnostic failure (CLI) or process exit code from `run`.

## Testing strategy

Tests live primarily in `compiler/src/lib.rs` (integration) and per-module unit tests.

| Category | Examples |
|----------|----------|
| Lexer | token spans, comments, identifier spellings |
| Parser | precedence, if/else, struct decls |
| Typeck | mutability, definite return, duplicates |
| Backend | LLVM IR snapshots with pinned triple |
| Diagnostics | stable codes, span accuracy |
| LSP | symbol index, hover content |
| Pipeline | no C backend strings in compile/llvm sources |

Run:

```bash
cargo test --manifest-path compiler/Cargo.toml
```

## Platform notes

### Windows

`llvm_windows_shim.rs` provides stub symbols for LLVM target initialization entry points missing from some Windows LLVM installer layouts.

Set `LLVM_HOME` when LLVM is not at `C:\Program Files\LLVM`.

## Extension points (post-v0.1)

| Area | Likely change |
|------|----------------|
| `typeck.rs` | Name resolution, struct types in signatures |
| `backend/llvm.rs` | Struct layout, `str` lowering, ABI details |
| `parser.rs` | New statements (loops, match) |
| `lsp.rs` | Cross-file index when imports resolve |
| `compile.rs` | Multi-file driver, artifact layout |

Keep frontend and backend rejection lists in sync when adding types ŌĆö maintain the `check` vs `emit-llvm` contract or close the gap intentionally.
